# Codex Superpowers — 代码审查质量规范

## 核心原则

Codex 在审查每份代码时，必须逐项检查以下维度：

## 检查清单

### 1. 测试覆盖
- [ ] 新增逻辑有对应的单元测试吗？
- [ ] 边界条件有测试吗？（空值、极限值、异常输入）
- [ ] 修改没有把已有的测试弄挂？
- [ ] 测试本身的质量合格吗？（不是单纯为了覆盖率写的 dummy test）

### 2. 错误处理
- [ ] 外部调用（API、数据库、文件系统）有 try/catch 或错误传播吗？
- [ ] 用户输入有校验吗？（SQL 注入、XSS、参数越界）
- [ ] 失败时有合理的 fallback 或错误提示吗？

### 3. 安全
- [ ] 涉及用户数据的地方有权限检查吗？
- [ ] API key、密钥、token 没有硬编码在代码里？
- [ ] 日志里没有打印敏感信息？

### 4. 代码质量
- [ ] 函数/方法职责单一？（不做「顺便改一下」的多余改动）
- [ ] 变量命名清晰？
- [ ] 没有死代码（注释掉的代码块、未使用的 import）？
- [ ] 复杂逻辑有注释说明为什么这么写？

### 5. 改动范围
- [ ] 改动只涉及任务包指定的范围？没有乱改不相干的文件？
- [ ] 配置文件变更是否经过确认？

## 输出格式

审查结论按 P0/P1/P2 分级：
- **P0（必须修）**：功能错误、安全漏洞、测试缺失
- **P1（建议修）**：代码异味、命名不合理、缺少注释
- **P2（可选）**：风格偏好、小优化

## 不可接受的行为

- ⛔ 不审查直接放行
- ⛔ 只说「看起来没问题」不列具体依据
- ⛔ 对 P0 问题只用「建议修改」的语气
